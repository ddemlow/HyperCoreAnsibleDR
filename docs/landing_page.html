The Problem

Manual DR failover fails when you need it most

Replication protects your data. But cloning VMs and bringing them online still requires a human — and humans aren't available, alert, or fast at 3am.

Without automation

  • Someone must notice the outage
  • Confirm it's real, not a blip
  • Log into the DR cluster manually
  • Find and clone each replicated VM
  • Power them on in the right order
  • Verify applications are running
  • Minutes to hours of downtime

With this playbook

  • Runs on a 5-minute schedule automatically
  • Uses multiple signals to confirm real failure
  • Clones all replicated VMs in one pass
  • Preserves MAC addresses for network continuity
  • Idempotent — safe to re-run indefinitely
  • Detection to VMs running in under 2 minutes
  • Works while everyone is asleep

How It Works

Multi-signal detection. Conservative by design.

The playbook never acts on a single indicator. Multiple independent checks must all agree before failover is triggered — preventing false positives during network blips or planned maintenance.

1

Check replication connection status

Queries the HyperCore API via remote_cluster_info. If the connection shows ESTABLISHED, the primary is reachable.

Exit — no failover needed
2

Ping all primary cluster node IPs

Node IPs are discovered dynamically from the replication record — no hardcoding. Each IP is pinged with configurable retries. If any node responds, the primary is still up.

Exit — primary nodes reachable
2b

Optional application-alive TCP check

Configure an application endpoint at the primary site. If it responds on the configured port, failover is aborted — even if the replication link looks broken.

Exit — application still responding
3

Verify DR cluster health

Pings all local DR cluster node IPs via node_info. A degraded DR cluster is not a safe target — abort rather than risk a bad failover.

Abort — DR cluster not healthy
4

Idempotency check

Checks for VMs already tagged with DR-failover. If they exist, failover has already run — exit cleanly. Safe to leave the schedule running indefinitely.

Exit — already failed over
5–6

Clone + power on replicated VMs

Clones every VM with a replication_source_vm_uuid, preserving MAC addresses for network identity continuity. Powers on all cloned VMs.

✓ DR Failover complete

Quick Start

Up and running in 30 minutes

No prior Ansible experience required. See the full Getting Started guide for detailed instructions including Windows (WSL2) setup. 

# 1. Install Ansible
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
ansible-galaxy collection install -r requirements.yml

# 2. Configure inventory (DR cluster IP + credentials)
cp inventory/inventory.example.yml inventory/inventory.yml
# edit inventory/inventory.yml

# 3. Test connectivity
ansible -i inventory/inventory.yml all \
  -m scale_computing.hypercore.cluster_info

# 4. Schedule (every 5 minutes via cron)
crontab -e
*/5 * * * * cd /opt/hypercore-ansible-dr && \
  .venv/bin/ansible-playbook -i inventory/inventory.yml \
  automated_dr_failover.yml >> /var/log/dr-failover.log 2>&1

Features

Built for production HyperCore environments

🔍

Dynamic node discovery

Primary cluster node IPs are fetched from the replication record — no hardcoded addresses to maintain.

🔁

Fully idempotent

Safe to run on any schedule indefinitely. After failover, re-runs detect existing DR VMs and exit cleanly.

🌐

Multiple source clusters

Monitor all remote clusters or a specific named one. Node IPs from all disconnected clusters are checked.

🏷️

VM exclusion tags

Tag infrastructure VMs (DNS, NTP, monitoring) with dr-exclude to skip them during failover.

🔒

MAC address preservation

Clones inherit the original VM's MAC addresses, maintaining network identity across DHCP and ARP.

🛟

Partial failure recovery

Block/rescue handles mid-failover errors cleanly, reporting exactly which VMs were cloned and which weren't.

⚙️

Fully configurable

All detection parameters are variables with sensible defaults. Override at runtime with -e.

🧪

Automated tests included

15 unit tests validate all decision logic using mocked data — runnable without a live cluster.

FAQ

Frequently asked questions

Will it trigger a false failover during a network blip?

No. The playbook requires both the HyperCore replication connection to show as disconnected and all primary cluster node IPs to fail ping after multiple retries. A brief network interruption will not meet both conditions simultaneously. You can further tune ping_retries and ping_delay for extra confidence.

What happens if I run the playbook again after failover?

Nothing. The playbook detects VMs already tagged with DR-failover and exits cleanly. It is designed to run on a continuous schedule — before, during, and after any failover event.

Do I need Ansible experience to use this?

No. The Getting Started guide walks through everything from scratch, including Python setup for Windows (WSL2) and macOS users.

Can I use this with multiple source clusters?

Yes. The playbook monitors all remote cluster connections by default. Set source_cluster_name to limit monitoring to a specific cluster. Note: because the HyperCore API on the target does not expose which source cluster a replicated VM came from, all replicated VMs are cloned when any confirmed-down cluster is detected.

Does it support failback?

Not yet. Failback (returning VMs to the primary after it's restored) requires manual steps currently. It is documented in IMPROVEMENTS.md as a planned enhancement.

Can I exclude certain VMs from failover?

Yes. Set dr_exclude_tag to a tag name and apply that tag to any VM on the DR cluster that should be skipped — for example, shared infrastructure like DNS or NTP servers that run independently at both sites.
